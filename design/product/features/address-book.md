# Address Book

状态：Accepted
更新时间：2026-03-13
已回写：[`prd.md`](../prd.md)

## 为什么有这份文档

这份文档用于定义地址簿功能的需求边界，并补充正式 PRD 未展开的 feature 说明。

当前钱包产品已经具备多链资产查询和转账能力，但在“收款目标复用”这件事上仍然依赖直接输入原始地址。随着支持资产变多，直接输入地址会持续增加以下问题：

- 常用收款目标无法复用
- Agent 需要重复记忆或传入长地址
- 人工和 Agent 在转账时更容易出现输错地址的风险
- 同一联系人在不同链上的地址关系无法被产品层表达
- 在真实使用场景中，用户会直接向 Agent 询问“某某地址是谁的”，而 Agent 难以判断该地址是否已经对应已知联系人

因此，地址簿的核心目标是提供一个面向转账使用的通讯录能力，让 Agent 既能以名称复用常用收款目标，也能在面对精确地址时反查该地址在地址簿中对应了哪些已知联系人。

同时，考虑到本产品是面向 Agent 使用的钱包，地址簿能力必须首先服务于 Agent 的日常使用。
Owner Mode 中对地址簿的支持，来自钱包产品已有的全局需求：只要 Agent 能执行的操作，Owner 也应能在 UI 中执行，而不是因为地址簿本身单独以 Owner 为主语来定义。

## 核心诉求

本次地址簿功能的核心诉求如下：

1. 形成稳定的名称到地址映射
2. Agent 在转账操作中可以直接使用名称
3. Agent 具备完整的新增、查看、修改和删除能力
4. 当同一个 `name` 在不同链上有不同地址时，系统能够根据转账链路自动映射到正确地址
5. Agent 能列出当前已有联系人
6. Agent 在需要时能一次性查看全部联系人的完整信息
7. Agent 在面对精确地址时，能反查该地址在地址簿中匹配到哪些联系人

## 产品定位

地址簿是钱包产品中的“联系人与收款目标管理能力”，而不是：

- 权限系统
- 风控系统
- 白名单系统
- 联系人社交系统

它的职责是：

- 管理可复用的收款目标名称
- 维护名称与不同链地址之间的映射关系
- 在转账时根据链上下文自动解析目标地址
- 在面对精确地址时返回地址簿中的匹配联系人

它的产品属性应被明确为：

- 一个共享的名称映射层
- 一个首先服务于 Agent 的目标管理能力
- 一个因全局接管需求而同步暴露给 Owner Mode 的辅助能力
- 一个服务于转账体验和目标复用的辅助能力

## 核心模型

### 1. 地址簿以名称为主视角

地址簿的核心对象是一个“联系人名称条目”，例如：

- `bob`
- `exchange`
- `team-wallet`

用户和 Agent 在使用地址簿时，首先操作的是这个名称，而不是底层原始地址。

### 2. 一个名称可以绑定多链地址

同一个名称可以在不同链上分别绑定地址。例如：

- `bob`
  - Nervos 地址：`ckb...`
  - Ethereum 地址：`0x...`

这意味着地址簿的产品心智不是“一个名称只对应一条地址”，而是“一个名称对应一组按链划分的地址映射”。

### 3. 同一名称在同一链下只允许一个地址

为了保证“转账时可以直接使用名称并自动映射”这件事始终成立，本期需求边界定义为：

- 同一个名称在同一条链下，只允许存在一个地址映射

例如：

- `bob` 可以同时有一个 Nervos 地址和一个 Ethereum 地址
- `bob` 不能同时有两个 Ethereum 地址

原因很直接：

- 如果同一链下允许多个地址，“直接使用名称转账”就会产生歧义
- 地址簿会从“自动映射”退化成“还要额外选地址”，不符合本次功能的核心目标

如果未来确实存在同一链下多个目标地址的业务需求，应通过不同名称表达，例如：

- `bob-main`
- `bob-hot`
- `bob-safe`

而不是在同一个名称下挂多个同链地址。

同时，本期不限制同一条链下的同一个地址被多个名称复用。

例如：

- `bob` 和 `bob-wallet` 可以同时记录同一个 Ethereum 地址

地址簿在这种情况下应保留这些记录，而不是强制要求地址唯一。

也就是说：

- 本期不约束同一个地址挂在多个联系人下

### 4. 映射按链上下文自动解析

当转账目标传入的是名称而不是原始地址时，系统应根据本次转账的链自动解析地址。

当前映射规则应为：

- Nervos 上的转账：使用该名称绑定的 Nervos 地址
- Ethereum 上的 ETH / USDT / USDC 转账：统一使用该名称绑定的 Ethereum 地址

这一定义意味着：

- `ETH`
- `USDT`
- `USDC`

虽然是不同资产，但它们在地址簿中共用同一个 Ethereum 地址映射。

## 支持行为

### 1. Agent 是地址簿能力的主语

地址簿功能的主要需求主语应是 Agent，而不是 Owner。

本次 feature 的核心要求是：

- Agent 能使用地址簿
- Agent 能管理地址簿
- Agent 能在转账时直接使用地址簿名称

### 2. Agent 管理地址簿

由于本产品的首要服务对象是 Agent，地址簿必须向 Agent 提供完整的新增、查看、修改和删除能力。

Agent 应支持：

- 创建地址簿名称条目
- 为名称新增链地址映射
- 编辑名称或链地址映射
- 删除名称条目
- 删除某条链地址映射
- 查看地址簿列表
- 按名称搜索联系人
- 查看单个联系人的完整信息
- 一次性查看全部联系人的完整信息
- 新增或修改联系人时，系统必须校验所提供地址是否合法

这意味着地址簿不能被定义为“仅 Owner 配置、Agent 只读”的能力。

### 3. Owner Mode 的支持来自既有全局需求

Owner Mode 之所以也需要支持地址簿，不是因为本次地址簿功能本身以 Owner 为核心主语，而是因为当前钱包产品已经明确存在以下全局规则：

- 只要 Agent 能执行的操作，Owner 也应能在 UI 中执行

因此，地址簿在 Owner Mode 中应具备与 Agent 对应的可操作能力，用于人工接管和手动操作。

### 4. Owner Mode 管理地址簿

Owner Mode 也应支持完整的新增、查看、修改和删除能力。

Owner 应支持：

- 查询地址簿列表
- 新增地址簿
- 修改地址簿
- 删除地址簿
- 按名称搜索地址簿
- 查看指定名称的完整信息
- 查看全部联系人的完整信息
- 在转账时直接使用名称

### 5. 地址簿是共享数据，不是权限边界

地址簿应被定义为共享数据层，而不是权限隔离层。

这意味着：

- Agent 创建的地址簿项，Owner 可以查看和修改
- Owner 创建的地址簿项，Agent 也可以查看和使用
- 地址簿本身不承载“谁创建就只属于谁”的权限语义
- 是否允许转账，不由地址簿决定，而由钱包其他规则决定

### 6. 转账时支持两种目标输入方式

本期转账目标应同时支持：

- 原始地址
- 地址簿名称

这可以保证：

- 现有原始地址输入能力不被破坏
- 地址簿作为增强能力逐步接入

### 7. 地址簿需要同时支持列表、详情与全量查看

为了让 Agent 能真正理解当前有哪些联系人，本期地址簿的读取能力应至少分为三层：

- 联系人列表
- 单个联系人详情
- 全部联系人完整信息
- 按精确地址查询匹配联系人

其中：

- 联系人列表应以 `name` 为主视角返回联系人摘要
- 联系人列表不按链拆成多行展示
- 联系人列表至少应包含名称、已记录的链和备注
- 单个联系人详情应返回该名称下的完整多链地址信息
- 全部联系人完整信息应一次性返回当前全部联系人的完整多链地址信息
- 按精确地址查询应返回该地址在地址簿中匹配到的全部联系人
- 匹配到多少联系人，就返回多少联系人

### 8. 按精确地址查询的实际意义

在真实使用场景中，用户会直接向 Agent 询问：

- `0x... 这个地址是谁的？`

因此，按精确地址查询能力的意义，不是判断链上真实所有者是谁，而是帮助 Agent 理解：

- 这个精确地址是否已经出现在地址簿中
- 这个精确地址当前关联了哪些已记录联系人
- 当用户直接给出一个地址时，Agent 是否能够用已有联系人信息继续完成后续判断或操作

因此，这项能力回答的是：

- 这个地址在地址簿里对应哪些联系人

而不是：

- 这个地址在链上真实属于谁

当同一个地址被多个名称复用时，按精确地址查询应返回全部匹配名称。匹配到多少联系人，就返回多少联系人，而不是强行只返回一个结果。

### 9. 名称解析失败时必须明确报错

当使用名称发起转账时，如果当前链下无法找到对应映射，系统必须明确失败。

至少应覆盖以下情况：

- 名称不存在
- 名称存在，但当前链下未绑定地址

地址簿不允许在解析失败时静默降级为其他链地址，也不允许模糊匹配。

## 在哪些行为中使用

本期建议地址簿仅用于与“收款目标”直接相关的行为，不扩展到所有钱包动作。

### 1. 转账

这是地址簿的核心使用场景，必须支持：

- Agent 转账时直接使用名称
- Owner Mode 转账时直接使用名称
- 转账确认或结果中展示“名称 + 实际地址 + 所属链”

### 2. Agent 的地址簿管理与查询

为了让 Agent 真正把地址簿作为目标管理层使用，本期必须支持：

- Agent 创建地址簿项
- Agent 查询地址簿列表
- Agent 搜索地址簿名称
- Agent 修改地址簿项
- Agent 删除地址簿项
- Agent 查看地址簿详情
- Agent 一次性查看全部地址簿完整信息
- Agent 通过精确地址查询匹配联系人
- Agent 新增或修改联系人时，地址需要通过合法性校验

### 3. Owner Mode 下的对应操作支持

由于 Owner Mode 需要承接所有 Agent 已暴露操作，本期也应支持：

- 列表查看
- 名称搜索
- 精确地址查询
- 条目详情查看
- 手动新增和编辑
- 手动删除
- 新增或修改联系人时，地址需要通过合法性校验

### 4. 不扩展到签名

本期不建议把地址簿扩展到签名、消息认证等非收款目标场景。

原因是：

- 地址簿的核心价值在于“转账目标管理”
- 如果一开始覆盖过多场景，会模糊产品边界

## 最小信息范围

本期地址簿条目建议至少包含以下信息：

- `name`
- 按链记录的地址
- `note`

其中：

- `name`：收款目标名称
- 按链记录的地址：该联系人在不同链上的地址信息
- `note`：选填备注

## 约束与边界

### 1. 名称必须可稳定解析

地址簿功能的核心价值是“用名称替代原始地址”。因此，名称必须具备稳定解析能力。

本期建议约束：

- 名称在地址簿内应唯一
- 同一名称在同一链下只允许一个地址
- 同一地址在同一链下可以被多个名称记录
- 新增或修改联系人时，地址必须是对应链的合法地址

### 2. 不做复杂联系人模型

本期不把地址簿设计成复杂通讯录系统。

不做的内容包括：

- 联系人分组
- 最近联系人
- 批量导入导出
- 链上标签同步
- 自动从历史转账生成地址簿

### 3. 不做地址簿白名单风控

本期地址簿只是名称映射层，不把它和白名单风控绑定。

也就是说：

- 地址簿中的地址不自动等于“允许地址”
- 地址簿外的地址也不自动等于“禁止地址”
- Agent 拥有地址簿的新增、查看、修改和删除能力，也不等于 Agent 获得额外转账权限

是否引入白名单能力，应作为后续独立需求讨论。

## 已确认的正式需求方向

本次已确认的正式需求，至少应落成以下几类正式需求：

1. 地址簿名称条目及多链映射能力
2. Agent 的地址簿新增、查看、修改和删除能力
3. Agent 的联系人列表、单个详情、全量查看与按精确地址查询能力
4. Owner Mode 对 Agent 地址簿能力的对应 UI 支持
5. 转账时支持地址簿名称作为目标输入
6. 按链自动解析地址簿名称
7. 按精确地址返回匹配联系人的能力定义
8. 联系人地址合法性校验要求
9. 解析失败时的明确错误返回
10. 地址簿作为共享名称映射层的产品定义
11. 地址簿需求主语以 Agent 为主，Owner 支持来自既有全局接管需求
