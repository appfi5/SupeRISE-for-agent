# Agent 信用钱包产品需求文档（PRD）

> 状态：Draft
> 更新时间：2026-03-13
> 版本：v2.0

## 1. 文档说明

本文档用于定义 Agent 信用钱包项目的产品需求边界，明确本期要解决的问题、目标用户、核心场景、功能范围与验收标准。

本文档只描述产品需求本身，不讨论技术方案、系统架构、实现方式或技术选型。
本文档只记录已经确认并生效的正式需求，不记录待讨论项或未定结论。

## 2. 项目背景

当前项目中的钱包相关能力已经具备一定基础，但缺少一个明确、统一且面向 Agent 场景的产品定义。

现阶段主要问题包括：

- 钱包能力没有围绕 Agent 使用场景进行独立定义
- 用户不知道应如何安全地把资产授权给 Agent 使用
- Agent 缺少一个稳定、明确、可持续扩展的钱包入口
- 钱包能力虽已可用，但尚未形成清晰的产品边界与使用心智

因此，本次项目的核心不是建设一个通用钱包，而是定义并交付一个专门为 Agent 使用的单账户信用钱包产品。

## 3. 问题定义

本次项目要解决的核心问题是：

1. 如何给 Agent 一个可独立使用的钱包，而不要求用户逐笔干预
2. 如何让用户在授予 Agent 完整操作能力的同时，仍能控制整体风险敞口
3. 如何通过 Owner Mode UI 让人类用户一键完成 Agent 可执行的操作
4. 如何做到无需任何 setup，程序启动后即可直接使用
5. 如何在无需前置人工确认的情况下，为后续人类介入保留 Owner 入口
6. 如何让单一私钥同时承载多链资产使用场景
7. 如何让 Agent 通过名称管理并复用常用收款目标
8. 如何让 Agent 在面对精确地址输入时识别已知联系人

本项目的基本判断是：

- Agent 不应直接接触私钥本体
- Agent 应可自由使用 server 已暴露的钱包能力
- 产品形态应明确为单账户、单私钥，而不是多账户或多钱包体系
- 风险控制不依赖复杂权限拆分，而依赖资金隔离
- 用户应将该钱包理解为 Agent 的信用钱包，而不是主资产钱包

## 4. 项目目标

本期目标如下：

1. 定义并交付一个专门供 Agent 使用的钱包产品
2. 支持系统自动生成 Agent 钱包，降低用户使用门槛
3. 让 Agent 能通过 server 暴露的能力完成多链资产的核心操作
4. 提供 Owner Mode UI，让人类用户可一键完成 Agent 可执行的核心操作
5. 实现零 setup 启动，程序运行后默认即可直接使用
6. 提供默认 Owner 凭证，使人类用户在需要时可后置介入
7. 明确向用户传达该钱包的风险模型和使用边界
8. 在必要时允许用户手动导出私钥并接管该钱包
9. 在本期明确支持 Nervos 上的 CKB，以及 Ethereum 上的 ETH、USDT 和 USDC
10. 为后续更多链能力和更多业务场景保留产品扩展空间
11. 提供地址簿能力，让 Agent 能以名称管理并使用常用收款目标
12. 提供按精确地址查询能力，让 Agent 能识别地址簿中的已知联系人

## 5. 非目标

本期不解决以下问题：

- 不建设面向多租户的平台级钱包托管产品
- 不追求面向高价值资产的强安全托管能力
- 不在本期完成所有潜在链能力的完整交付
- 不以兼容旧入口、旧交互方式或旧结构为目标
- 不建设复杂的企业级运营后台
- 不建设多账户或多钱包管理体系

## 6. 产品定位

本次重构后的钱包产品定位为：

- 一个专门供 Agent 使用的钱包产品
- 一个由用户出资、由 Agent 通过 server 能力自主使用的信用钱包
- 一个单账户、单私钥、面向多链资产使用场景的钱包产品
- 一个通过资金隔离而不是权限收紧来控制风险的产品
- 一个同时提供 Agent 调用入口和 Owner Mode UI 的统一钱包产品

该产品的职责是：

- 自动生成并管理 Agent 专用钱包
- 提供基于单一私钥的多链身份查询能力
- 提供签名能力
- 提供多链资产转账能力
- 提供地址簿能力，用于管理和复用常用收款目标
- 提供 Owner Mode UI，支持人类用户直接操作钱包
- 提供必要的钱包状态查看能力
- 在特殊情况下允许用户手动接管钱包
- 为其他产品能力提供统一的钱包入口

该产品不负责：

- 承载复杂业务流程本身
- 替代上层业务系统的业务规则
- 提供高价值资产托管承诺

## 7. 目标用户与角色

### 7.1 核心用户

本期涉及两类核心用户：

- Owner：在需要人工介入时，通过默认凭证或更新后的凭证进入 Owner Mode，查看和操作钱包，并在必要时手动导出或恢复私钥
- Agent 使用方：通过标准化入口调用钱包能力，以完成查询、签名、转账等任务

### 7.2 角色定义

为保证需求边界清晰，本期定义以下角色：

- `Owner Mode`：由 Owner 通过默认凭证或更新后的凭证进入的高权限操作模式，可执行所有 Agent 已暴露能力，并执行恢复、导出等更高权限操作
- `Agent`：钱包的主要使用者，不接触私钥本体，但可自由使用 server 已暴露的钱包能力
- `集成方`：后续接入钱包能力的其他模块或产品，不直接定义钱包规则，只调用标准化能力

## 8. 核心使用场景

### 8.1 场景一：创建 Agent 信用钱包

作为 Agent 使用方，我希望程序在启动时默认生成并启用一个 Agent 专用钱包，而不要求先完成任何 setup，以便系统运行后即可直接开始使用。

### 8.2 场景二：程序启动后默认直接可用

作为 Agent 使用方，我希望程序在运行后默认即可使用当前钱包，而不是先经过 Owner 确认或初始化流程，以便安装门槛尽可能低。

### 8.3 场景三：以有限注资控制风险

作为 Owner，我需要只向 Agent 钱包放入可承受风险范围内的资金，而不是将全部资产放入该钱包，以便将 Agent 的操作风险控制在可接受范围内。

### 8.4 场景四：Agent 自主执行链上操作

作为 Agent，我需要能够通过 server 暴露的能力完成 Nervos 上的 CKB 操作，以及 Ethereum 上的 ETH、USDT 和 USDC 操作，以便独立完成链上任务，而不需要用户逐笔干预。

### 8.5 场景五：Owner Mode 通过 UI 一键完成核心操作

作为 Owner，我需要通过 Owner Mode UI 一键完成查询、签名、转账等核心操作，以便在需要人工介入时能够直接接管 Agent 的执行任务。

### 8.6 场景六：特殊情况下手动接管

作为 Owner，我需要在特殊情况下手动导出私钥或手动导入恢复私钥并自行操作该钱包，以确保当 Agent 无法继续完成任务时，资金仍可被我接管。

### 8.7 场景七：Owner Mode 查看钱包状态与风险

作为 Owner，我需要通过 Owner Mode UI 查看 Agent 钱包的身份、状态和风险提示，以便判断当前风险敞口是否仍在预期范围内。

### 8.8 场景八：使用默认 Owner 凭证介入

作为 Owner，我希望程序启动后能提供一个默认 Owner 凭证，以便我在需要人工介入时可以直接进入 Owner Mode，而不是必须在首次运行时完成 setup。

### 8.9 场景九：通过恢复流程导入钱包

作为 Owner，我需要在设备迁移、数据丢失或重装等特殊情况下，通过 UI 手动导入私钥来恢复钱包，以便重新取得该信用钱包的控制权。

### 8.10 场景十：Owner 配置币种限额

作为 Owner，我需要在 UI 中按币种分别设置日、周、月限额，以便限制 Agent 对不同资产的风险敞口。

### 8.11 场景十一：Agent 触发限额

作为 Agent，我不需要主动感知限额配置，但在触发限额时，我需要收到明确的 limit 信息，以便知道当前请求为什么被拦截。

### 8.12 场景十二：Agent 管理常用联系人

作为 Agent，我需要能够新增、查看、修改和删除常用联系人，以便持续维护可复用的收款目标。

### 8.13 场景十三：Agent 通过名称发起转账

作为 Agent，我需要在转账时直接使用联系人名称，而不是每次输入原始地址，以便更稳定地复用常用收款目标。

### 8.14 场景十四：Agent 根据精确地址识别已知联系人

作为 Agent，我需要在真实使用场景中回答用户“某某地址是谁的”这类问题，因此当用户或上游直接给出一个精确地址时，我需要查询该地址在地址簿中匹配到哪些联系人，以便理解这是不是一个已知收款目标。

## 9. 需求范围

### 9.1 本期范围内

本期 MVP 需要覆盖以下能力：

- 自动生成 Agent 钱包
- 手动导入私钥恢复钱包
- 当前钱包查看
- 多链身份信息查看
- 消息签名
- 已支持币种的余额查询
- CKB 转账闭环
- ETH 转账闭环
- USDT 转账闭环
- USDC 转账闭环
- 地址簿能力
- server 侧币种限额能力
- 私钥手动导出
- 面向 Agent 的钱包能力调用入口
- Owner Mode UI
- 面向 Owner 的钱包状态查看与风险说明
- Owner Mode 权限体系

### 9.2 本期范围外

以下内容不纳入本期 MVP：

- 多租户管理
- 企业级托管能力
- 面向高价值资产的高级安全能力
- 所有链能力的完整覆盖
- CKB 之外的其他 CKB 资产
- 除 ETH、USDT 与 USDC 之外的其他 Ethereum 资产
- 复杂通讯录能力
- 其他业务模块的功能重构
- 复杂的企业级运营后台

## 10. 产品原则

本期需求定义遵循以下原则：

1. Agent 优先：钱包的首要服务对象是 Agent，而不是人工逐笔操作
2. 风险敞口可控优先：用户应通过有限注资控制风险，而不是依赖复杂权限拆分
3. 人工可接管优先：只要 Agent 能执行的核心操作，Owner 也应能通过 UI 一键完成
4. 单账户优先：本期产品默认只有一个钱包，不引入多钱包心智
5. 场景闭环优先：优先保证关键使用场景能真正完成
6. 可理解优先：需求描述应面向用户价值和业务动作，而不是实现细节
7. 可接管优先：即使 Agent 是主要操作者，Owner 也必须保留最终接管能力

## 11. 功能需求

### 11.1 钱包创建与管理

#### FR-1 首次运行自动生成 Agent 钱包

系统应支持在程序首次运行时默认自动生成一个供 Agent 使用的钱包，作为后续所有钱包操作的前置条件。

要求：

- 钱包应由系统自动生成，不要求用户手动提供私钥
- 默认生成应在启动阶段完成，而不是要求用户先理解复杂创建流程
- 生成成功后应立即可用于身份查询、签名和转账
- 钱包生成不应以 Owner 确认为前置条件
- 生成失败时应返回明确失败原因

#### FR-2 手动导入私钥恢复钱包

系统应支持 Owner 在 Owner Mode UI 中手动导入私钥，用于恢复既有钱包。

要求：

- 手动导入的定位应明确为恢复流程，而不是默认创建方式
- 恢复成功后，导入钱包应立即可用于身份查询、签名和转账
- UI 中必须明确提示该操作的使用场景和风险
- 导入失败时应返回明确失败原因

#### FR-3 零 Setup 启动

系统应支持在无人工 setup 的前提下直接启动并启用钱包能力。

要求：

- 程序运行后应默认直接可用，而不是要求先完成 Owner 初始化
- Agent 使用不应依赖 Owner 预先登录或确认
- 钱包启用不应要求用户先进入 UI 完成初始化流程
- 对外文档应可支持 Agent 按说明自行安装后直接使用

#### FR-4 默认 Owner 凭证

系统应在启动后提供一组默认 Owner 凭证，供人类用户在需要时进入 Owner Mode。

要求：

- 默认 Owner 凭证应在程序可访问的位置被明确提供给用户
- 用户可以使用默认 Owner 凭证进入 Owner Mode
- 用户在进入 Owner Mode 后应可修改该默认凭证
- 默认 Owner 凭证的存在不应阻塞 Agent 正常使用
- 产品文案中应明确说明默认 Owner 凭证的用途和风险

#### FR-5 当前钱包查看
系统应支持查看当前唯一钱包的基础信息。

要求：

- 返回当前钱包的基础标识信息
- 当系统中尚未生成或恢复钱包时，应返回明确状态
- 产品中不引入多钱包列表心智

#### FR-6 查看多链身份信息

系统应支持查看当前钱包在不同链上的身份信息。

要求：

- 可查看钱包基础元数据
- 可查看同一私钥对应的 CKB 身份信息
- 可查看同一私钥对应的 ETH 身份信息
- 当钱包不存在时，应返回明确提示

#### FR-7 单钱包识别

系统应能明确识别当前唯一钱包，以便所有操作都围绕同一钱包进行。

要求：

- 查询当前钱包时应返回唯一钱包信息
- 若不存在钱包，应返回明确状态
- 产品中不要求用户在多个钱包之间进行选择

#### FR-8 Owner Mode UI

系统应提供一个 Owner Mode UI，供 Owner 查看、确认、恢复和操作 Agent 钱包。

要求：

- Owner 完全通过 UI 完成高权限操作，不要求使用命令行或其他技术入口
- Owner 可通过 UI 查看当前钱包和多链身份信息
- Owner 可通过 UI 触发 Agent 已具备的核心钱包操作
- Owner 可通过 UI 完成导出私钥和恢复导入私钥等高权限操作
- Owner 可通过 UI 查看和修改默认 Owner 凭证
- UI 中的核心操作应尽量以一键方式完成，而不是要求用户拼装复杂参数流程
- 当操作失败时，UI 应返回明确结果和失败原因

### 11.2 身份与签名

#### FR-9 身份查询

系统应支持基于系统生成的钱包返回可用于识别该钱包的多链身份信息。

要求：

- 返回结果应足以让使用者确认钱包身份
- 返回信息应与后续签名、转账所使用的钱包保持一致
- 返回结果至少包括 CKB 身份信息和 ETH 身份信息

#### FR-10 消息签名

系统应支持使用指定钱包对给定消息完成签名。

要求：

- Agent 可使用指定钱包执行签名
- Owner 也可通过 UI 使用指定钱包执行签名
- 签名结果应可直接用于身份认证或授权场景
- 对无效请求、钱包不存在等情况，应返回明确错误

### 11.3 转账

#### FR-11 已支持币种能力一致性

凡是本期声明为“支持”的币种，必须同时具备余额查询能力和转账能力。

要求：

- 不允许出现只支持查询余额、不支持转账的币种
- 不允许出现只支持转账、不支持查询余额的币种
- 本规则适用于本期所有已支持币种

#### FR-12 CKB 余额与转账能力

系统应支持当前钱包对 Nervos 上的 CKB 进行余额查询和转账。

要求：

- Agent 可查询 CKB 余额
- Owner 也可通过 UI 查询 CKB 余额
- Agent 可执行 CKB 转账
- Owner 也可通过 UI 执行 CKB 转账
- 系统应返回明确的处理结果
- 失败时应返回可诊断的失败原因

#### FR-13 ETH 余额与转账能力

系统应支持当前钱包对 Ethereum 上的 ETH 进行余额查询和转账。

要求：

- Agent 可查询 ETH 余额
- Owner 也可通过 UI 查询 ETH 余额
- Agent 可执行 ETH 转账
- Owner 也可通过 UI 执行 ETH 转账
- 系统应返回明确的处理结果
- 失败时应返回可诊断的失败原因

#### FR-14 USDT 余额与转账能力

系统应支持当前钱包对 Ethereum 上的 USDT 进行余额查询和转账。

要求：

- Agent 可查询 USDT 余额
- Owner 也可通过 UI 查询 USDT 余额
- Agent 可执行 USDT 转账
- Owner 也可通过 UI 执行 USDT 转账
- 系统应返回明确的处理结果
- 失败时应返回可诊断的失败原因

#### FR-15 USDC 余额与转账能力

系统应支持当前钱包对 Ethereum 上的 USDC 进行余额查询和转账。

要求：

- Agent 可查询 USDC 余额
- Owner 也可通过 UI 查询 USDC 余额
- Agent 可执行 USDC 转账
- Owner 也可通过 UI 执行 USDC 转账
- 当前 testnet 合约地址为 `0xa704C2f31628ec73A12704fa726a1806613a30ae`
- 系统应返回明确的处理结果
- 失败时应返回可诊断的失败原因

#### FR-16 多币种转账闭环

系统应提供完整的多链转账闭环能力，而不是只完成局部步骤。

“转账闭环”在本期的定义为：

- Agent 提交转账请求
- 系统完成必要处理
- 系统返回明确的转账结果标识

要求：

- CKB 转账闭环必须可用
- ETH 转账闭环必须可用
- ETH 上的 USDT 转账闭环必须可用
- ETH 上的 USDC 转账闭环必须可用

### 11.4 限额能力

#### FR-17 币种独立限额能力

系统应在 server 侧为每个已支持币种提供独立限额能力。

要求：

- 限额应按币种独立配置
- 当前已支持币种 CKB、ETH、USDT、USDC 都必须具备独立限额能力
- 限额周期固定为日、周、月三档
- server 应负责执行限额校验，不能依赖 Agent 自行判断
- 限额重置应按 server 本地时区计算
- 日限额应在每天 00:00 重置
- 周限额应在每周一 00:00 重置
- 月限额应在每月 1 日 00:00 重置

#### FR-18 限额配置入口

系统应支持 Owner 通过 Owner Mode UI 对每个已支持币种设置限额。

要求：

- Owner 可分别设置每个币种的日限额、周限额和月限额
- 每个币种的三档限额互相独立
- 限额配置应作为 Owner Mode 中的正式能力存在
- Agent 不应具备查看或修改限额的能力
- Owner 自身操作不受该套币种限额约束

#### FR-19 限额触发反馈

当 Agent 触发限额时，系统应阻止本次请求，并返回明确的 limit 信息。

要求：

- Agent 在未触发限额前不需要主动感知限额配置
- Agent 在触发限额时，应收到明确的 limit 提示
- 返回内容应足以让 Agent 知道是哪个币种、哪个周期的限额被触发
- 限额反馈应由 server 返回，保持与其他错误返回一致

### 11.5 手动接管

#### FR-20 手动导出私钥

系统应支持 Owner 在必要时通过 UI 手动导出私钥，以确保钱包控制权可恢复。

要求：

- 私钥导出属于高风险操作
- 仅 Owner 可执行该操作
- 该操作必须具备明确的风险提示
- 该操作必须具备明确的操作记录

#### FR-21 Owner 查看钱包状态

系统应支持 Owner 通过 UI 查看 Agent 钱包的当前状态，以辅助其控制风险敞口。

要求：

- 状态展示应按链分组
- 当前需要展示的链为 CKB 和 ETH
- 每条链下只展示对应地址
- 资产展示应按链下资产分类
- 当前需要展示的资产为 CKB 链下的 CKB，以及 ETH 链下的 ETH、USDT 与 USDC
- 每种资产默认只展示余额
- 每种资产应可显示对应限额状态或限额配置入口
- 整体应以看板形式展示，不默认呈现过多技术或交易细节
- 当钱包不存在时，应返回明确提示

### 11.6 风险提示

#### FR-22 信用钱包风险说明

系统应明确告知用户：该钱包是 Agent 的信用钱包，Agent 不直接接触私钥，但可自由使用 server 已暴露的钱包能力。

要求：

- 产品文案中必须明确说明该钱包不适合存放全部资产
- 产品文案中必须明确说明用户应以有限注资方式控制风险敞口
- 产品文案中必须明确说明该钱包的安全边界来自资金隔离，而不是对已暴露能力的细粒度限制
- 产品文案中必须明确说明 Agent 无法直接获取或导出核心私钥
- 产品文案中必须明确说明系统默认可直接运行，Owner 介入不是前置条件
- 产品文案中必须明确说明交易详情不作为产品内默认展示内容，用户可通过地址自行使用区块浏览器查看

### 11.7 Agent 调用能力

#### FR-23 Agent 统一入口

系统应提供面向 Agent 的统一钱包调用入口，使 Agent 能以标准方式访问钱包能力。

要求：

- Agent 可完成查询、签名、转账等核心操作
- Agent 是该钱包的主要操作者，应能自由使用 server 已暴露的核心资金操作能力
- Agent 调用的钱包能力边界应保持稳定一致

#### FR-24 Agent 可用能力范围

本期 Agent 入口至少需要覆盖以下能力：

- 查询当前钱包
- 查询多链身份信息
- 查询 CKB 余额
- 查询 ETH 余额
- 查询 USDT 余额
- 查询 USDC 余额
- 发起消息签名
- 发起 CKB 转账
- 发起 ETH 转账
- 发起 ETH 上的 USDT 转账
- 发起 ETH 上的 USDC 转账

说明：

- 私钥导出不应作为 Agent 默认开放能力
- Agent 不应具备查看限额配置或修改限额的能力
- Agent 不应直接接触私钥本体
- 只要能力已经由 server 暴露给 Agent，默认不做额外细粒度限制

### 11.8 Owner Mode 权限

#### FR-25 Owner Mode 高权限能力

系统应提供一组高于 Agent 的 Owner Mode 能力，用于完成钱包管理、恢复和人工接管。

要求：

- Owner 应具备执行所有 Agent 已暴露能力的权限
- Owner 应具备 Agent 默认不具备的高权限操作能力
- 至少“恢复导入私钥”“查看状态”“导出私钥”“限额设置”属于 Owner 专属能力

#### FR-26 Owner 后置介入流程

系统应定义一个清晰的 Owner 后置介入流程，使人类用户在需要时再进入 Owner Mode，而不是在首次启动前完成 setup。

要求：

- Owner 不应被要求在首次启动前完成 setup
- 默认 Owner 凭证应支持 Owner 在需要时随时进入 Owner Mode
- Owner 首次介入后应可立即执行查看、修改凭证、恢复导入和导出等操作
- 若 Owner 选择恢复钱包，则恢复后的钱包应直接替代默认生成钱包成为当前唯一钱包
- 若介入流程存在关键风险提示，必须在 UI 中显式展示

### 11.9 地址簿

#### FR-27 地址簿联系人能力

系统应提供地址簿能力，用于以名称管理常用收款目标。

要求：

- 地址簿应以联系人名称为核心对象
- 同一个联系人名称可分别记录 Nervos 地址和 Ethereum 地址
- 地址簿中的联系人名称应唯一
- 同一个联系人名称在同一条链下只允许记录一个地址
- 同一条链下的同一个地址可以被多个联系人名称记录
- 本期不引入联系人分组、最近联系人或批量导入导出等复杂通讯录能力

#### FR-28 Agent 地址簿管理能力

系统应支持 Agent 管理地址簿，使其能够持续维护常用收款目标。

要求：

- Agent 可新增联系人
- Agent 可修改联系人名称、地址和备注
- Agent 可删除联系人
- Agent 可查看联系人列表
- Agent 可按名称搜索联系人
- Agent 可查看单个联系人的完整信息
- Agent 可一次性查看全部联系人的完整信息

#### FR-29 按精确地址查询匹配联系人

系统应支持 Agent 通过精确地址查询地址簿中的匹配联系人。

要求：

- 查询应以精确地址为条件，不支持模糊匹配
- 查询结果应表达该地址在地址簿中匹配到哪些联系人
- 查询结果不应表达该地址在链上的真实所有权或真实归属
- 当同一地址被多个联系人名称记录时，查询结果应返回全部匹配联系人
- 当地址簿中没有匹配联系人时，系统应明确返回未匹配结果

#### FR-30 按链自动解析联系人名称

当转账目标使用联系人名称时，系统应按当前转账链自动解析到正确地址。

要求：

- Nervos 上的 CKB 转账应使用该联系人记录的 Nervos 地址
- Ethereum 上的 ETH、USDT 和 USDC 转账应统一使用该联系人记录的 Ethereum 地址
- 当联系人名称不存在时，系统应明确失败
- 当联系人存在但当前链下未记录地址时，系统应明确失败
- 解析失败时不允许降级为其他链的地址，也不允许模糊匹配

#### FR-31 转账时支持联系人名称

系统应支持在转账时使用联系人名称作为收款目标输入方式。

要求：

- 转账目标应同时支持原始地址和联系人名称两种输入方式
- Agent 可在转账时直接使用联系人名称
- Owner 也可通过 UI 在转账时直接使用联系人名称
- 当转账使用联系人名称时，结果中应明确展示所使用的联系人名称和实际地址

#### FR-32 Owner Mode 对应支持

系统应在 Owner Mode UI 中提供与 Agent 地址簿能力对应的操作支持。

要求：

- Owner 可查看联系人列表
- Owner 可按名称搜索联系人
- Owner 可按精确地址查询匹配联系人
- Owner 可查看单个联系人的完整信息
- Owner 可一次性查看全部联系人的完整信息
- Owner 可新增、修改和删除联系人
- Owner 可在转账时直接使用联系人名称

## 12. 权限与约束

### 12.1 角色权限要求

本期至少应区分两类权限：

- Owner Mode 权限
- Agent 运行权限

其中：

- Owner Mode 权限负责查看、修改凭证、恢复导入、导出、风险控制和人工接管
- Agent 运行权限负责使用 server 已暴露的多链签名、余额查询、转账和地址簿管理等能力

### 12.2 风险控制要求

本期虽然不追求高级托管安全能力，但必须明确告知使用者以下事实：

- 该产品属于 MVP 形态
- Agent 不直接持有核心私钥
- Agent 可自由使用 server 已暴露的钱包能力
- Owner Mode 权限高于 Agent，并可通过 UI 一键完成人工介入
- 系统默认可直接使用，Owner 介入属于可选能力
- 用户应只向该钱包放入可承受风险范围内的资产
- 币种限额由 Owner 配置，并由 server 执行
- Agent 不主动感知限额配置，仅在触发时收到 limit 信息
- Owner 自身操作不受币种限额约束
- 不应将其视为高价值资产的强安全托管方案

## 13. 非功能需求

### 13.1 易理解性

- 用户应能清楚理解钱包产品的职责边界
- 用户应能清楚理解“Agent 信用钱包”的风险模型
- 用户应能清楚理解该产品是单账户、单私钥、多链资产的钱包

### 13.2 一致性

- 不同入口对同一钱包能力的定义应保持一致
- 同一操作在不同入口中的结果语义应保持一致
- 钱包状态展示应保持统一看板口径，不因入口不同而暴露不同层级的链上细节

### 13.3 可扩展性

- 本期需求定义不能阻碍后续支持更多链能力
- 本期需求定义不能阻碍后续接入更多调用入口或业务场景
- 本期需求定义不应引入多钱包管理复杂度

### 13.4 可验收性

- 每项核心需求必须能被明确验证是否完成
- 不允许以模糊表述作为需求完成依据

## 16. 依赖与前提

本期需求成立基于以下前提：

- 当前已有的钱包相关能力可作为需求梳理基础
- 用户愿意接受“Agent 不接触私钥，但可自由使用已暴露能力”的产品边界
- 用户愿意以有限注资方式控制风险敞口
- Owner 需要有明确可用的 UI 入口完成查看、修改凭证、恢复、管理和接管操作
- 后续其他业务模块会以该钱包产品能力作为依赖，而不是继续自建钱包规则

## 17. 风险

### 17.1 当前已知风险

- 若用户未理解该钱包是信用钱包，容易错误存入过多资产
- 若管理入口过弱，用户在特殊情况下可能无法及时手动接管
- 若 Owner Mode 能力与 Agent 能力映射不完整，UI 会失去实际接管价值
- 若默认 Owner 凭证展示或修改路径不清晰，Owner 介入成本会过高
- 若单钱包模式与多链资产展示关系定义不清，用户容易误解钱包结构
- 若限额周期边界定义不清，Owner 对日/周/月限额的理解可能出现偏差
- 若联系人名称与链地址的映射规则不清，Agent 使用名称转账时容易出现误判
- 若联系人列表与完整信息的边界不清，Agent 难以高效理解当前可复用的收款目标
- 若按精确地址查询的含义不清，使用者容易把它误解为链上真实归属识别能力

## 18. 术语说明

- 钱包：由系统生成并供 Agent 通过 server 能力使用的单一资产控制单元
- Owner：该信用钱包的人类控制者，在需要时通过默认凭证或更新后的凭证进入 Owner Mode 完成恢复和接管操作的用户
- Owner Mode：Owner 使用的高权限操作模式，可执行 Agent 已暴露能力，并执行修改凭证、恢复导入、导出、限额设置等更高权限操作
- Agent：通过标准化入口调用钱包能力的自动化使用方
- 单账户钱包：本期只存在一个钱包和一把私钥，不引入多钱包或多账户管理
- 身份信息：用于识别钱包归属和确认钱包对象的信息
- 资产看板：按链分组、按资产展示余额的轻量视图，不默认展示交易明细和底层链上细节
- 币种限额：针对单个币种独立设置的日、周、月额度控制，由 Owner 配置并由 server 执行
- 地址簿：通过联系人名称管理多链收款目标的通讯录能力
- 联系人：地址簿中的单个名称条目，可记录不同链上的地址和备注
- 地址匹配查询：通过精确地址返回地址簿中的匹配联系人，不表示链上真实所有者
- 高风险操作：一旦执行可能影响资产控制权或系统安全边界的操作
- 信用钱包：用户只放入有限资金、允许 Agent 自主使用的专用钱包
- 多链资产：由同一私钥在不同链上映射出的资产使用场景，本期明确包括 Nervos 上的 CKB，以及 Ethereum 上的 ETH、USDT 和 USDC
- 转账闭环：从发起请求到获得明确转账结果的完整过程
