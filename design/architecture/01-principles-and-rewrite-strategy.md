# 01 重构原则与清理策略

## 1. 文档目的

本文档定义本次重构的实施原则、历史清理范围和禁止事项。它解决的问题不是“怎么改旧代码”，而是“为什么不能继续站在旧代码上演进”。

## 2. 重构性质

本次工作明确属于彻底重构，而不是增量改造。

正式定义如下：

- 不保留旧实现的兼容义务
- 不保留旧接口协议
- 不保留旧目录结构
- 不保留旧工程启动方式
- 不以旧代码可复用程度作为技术决策依据

判断依据：

- PRD 明确不以兼容旧入口、旧交互方式或旧结构为目标
- 历史实现的核心边界与新产品边界不一致
- 继续在旧实现上演进，会把旧的错误抽象、错误接口和错误部署模式带入新系统

## 3. 必须先清理的内容

开发新架构前，必须先完成以下清理：

### 3.1 旧服务端实现

- 清理旧 `.NET local server` 相关代码
- 清理旧 server 配置、安装脚本、运行脚本
- 清理旧 swagger、旧 endpoint 定义和旧认证逻辑

原因：

- 新架构服务端已定为 `NestJS 11`
- 旧 `.NET` 实现继续存在会制造双主实现假象

### 3.2 旧 Agent 入口

- 清理旧 Agent HTTP 接口
- 清理以 HTTP 为核心的 Agent 钱包调用说明
- 清理围绕旧接口命名形成的能力定义

原因：

- Agent 统一入口已经正式定为 `MCP`
- 同时保留 MCP 和 Agent HTTP 会制造协议分裂

### 3.3 旧 CLI 与旧脚本

- 清理与新架构无关的旧 CLI 命令
- 清理旧启动脚本、旧构建脚本、旧安装脚本
- 清理旧本地调试脚本与旧文档

原因：

- 本期主系统不是 CLI 产品
- CLI 不是当前 Agent 入口，也不是 Owner 入口

### 3.4 旧文档

- 清理与新架构冲突的旧设计文档
- 清理描述旧系统结构的 README、设计稿、部署说明
- 保留 PRD，重建正式架构文档集

原因：

- 一套系统只能有一份有效的技术真相来源

## 4. 新建优先原则

本期实现按“新建优先”推进：

- 新建 monorepo
- 新建 `NestJS` 服务
- 新建 `React + Vite` Owner UI
- 新建 `packages` 共享层
- 新建数据库 schema
- 新建密钥管理实现

禁止做法：

- 将旧服务端逐目录搬迁到新仓库结构
- 为了复用旧代码保留错误抽象
- 为了减少短期工作量保留旧接口

## 5. 架构约束的优先级

开发过程中出现冲突时，优先级按下面顺序执行：

1. `PRD`
2. 本文档集中的架构约束
3. 安全与密钥管理约束
4. 实现便利性
5. 历史代码可复用性

这意味着：

- 若旧实现与 PRD 冲突，删旧实现
- 若实现便利性与安全边界冲突，保留安全边界
- 若架构清晰度与短期编码速度冲突，保留架构清晰度

## 6. 首次开发前的清场清单

建议作为第一阶段任务执行：

1. 删除旧 server 目录与安装脚本
2. 删除旧 CLI 中与本期无关的钱包入口实现
3. 删除旧 Agent HTTP 说明与旧 swagger 文件
4. 删除过时的配置文件、运行脚本和调试脚本
5. 保留 `design/product/prd.md`
6. 建立新的 `apps/`、`packages/`、`design/architecture/`

## 7. 输出要求

清场完成后，仓库应满足以下状态：

- 仓库中只有一套新架构主实现
- Agent 入口只剩 `MCP`
- Owner 入口只剩 `HTTP + Web UI`
- 不再存在可能被误认成正式入口的历史实现
