# 07 安全、加密与密钥管理

## 1. 文档目的

本文档定义安全边界、私钥加密方案、密钥提供方式、部署侧控制权和禁止事项。它是整个系统中优先级最高的技术文档之一。

## 2. 安全边界声明

本系统是本地 MVP 信用钱包，不是高价值资产托管系统。

这意味着：

- 系统需要对私钥做正式加密管理
- 系统需要避免私钥在普通流程中暴露
- 系统不承诺抵御宿主机完全失陷
- 系统不采用 HSM、MPC、远程托管或企业级 KMS 作为本期前提

## 3. 密钥层次

本期采用双层密钥设计：

- `Wallet Private Key`
- `DEK`
- `KEK`

### 3.1 `Wallet Private Key`

- 用于推导 CKB 与 ETH 身份
- 用于完成签名和转账
- 绝不明文落盘

### 3.2 `DEK`

- 每个钱包生成一个随机 `DEK`
- 用来加密钱包私钥
- 与钱包数据一起持久化，但只能以被 `KEK` 包装后的形式保存

### 3.3 `KEK`

- 系统根密钥
- 用于包装 / 解包 `DEK`
- 不进入数据库
- 默认由部署侧提供

## 4. 加密算法

固定使用：

- `AES-256-GCM`

规则：

- `KEK` 长度固定 `32 bytes`
- `DEK` 长度固定 `32 bytes`
- 每次加密都使用独立 `iv`
- 保存 `tag`
- 若 `KEK` 来源是口令字符串，则先使用 `scrypt` 派生

## 5. 加密流程

### 5.1 创建钱包

1. 生成钱包私钥
2. 生成随机 `DEK`
3. 用 `DEK` 加密钱包私钥
4. 用 `KEK` 包装 `DEK`
5. 将私钥密文、包装后的 `DEK`、`iv`、`tag` 落库

### 5.2 使用钱包

1. 从存储层读取私钥密文与包装后的 `DEK`
2. 使用 `KEK` 解包 `DEK`
3. 使用 `DEK` 解密私钥
4. 在内存中完成签名或转账

### 5.3 导入钱包

1. 校验私钥输入
2. 重新生成 `DEK`
3. 用外部 `KEK` 包装 `DEK`
4. 替换当前钱包存储

## 6. `KEK` 提供策略

### 6.1 默认原则

- 应用不负责生成正式 `KEK`
- 应用只负责读取外部提供的 `KEK`
- 未读取到 `KEK` 时默认启动失败

### 6.2 支持的 provider

1. `docker-secret`
2. `file-path`
3. `env`

优先级：

1. `WALLET_KEK_PATH`
2. `WALLET_KEK`
3. 仅限开发模式的自动生成

### 6.3 为什么默认不用 env

因为环境变量泄漏面更大：

- 启动脚本可能落盘
- 进程环境可能被诊断工具读取
- CI/CD 与运行平台更容易复制传播

所以：

- `env` 只作为兼容模式
- 非默认正式方案

## 7. docker 与非 docker 策略

### 7.1 docker-compose

正式默认：

- 使用 `docker secret`
- 容器内读取 `/run/secrets/wallet_kek`

### 7.2 非 docker

正式默认：

- 使用 `WALLET_KEK_PATH`
- 指向受控文件
- 文件权限仅服务用户可读

### 7.3 开发模式

允许：

- 显式开启自动生成 `KEK`

限制：

- 仅本地开发
- 必须明确标记为非生产模式

## 8. 控制权说明

### 8.1 使用 docker secret 时

核心秘钥掌握在：

- 部署者
- 宿主机管理员

### 8.2 使用受控文件路径时

核心秘钥掌握在：

- 系统管理员
- 具备文件读取权限的部署人员

### 8.3 使用 env 时

核心秘钥掌握在：

- 运行环境管理者
- 可能读取环境变量的相关系统

因此 `env` 不作为正式默认。

## 9. 安全禁止事项

明确禁止：

- 私钥明文写入数据库
- 私钥明文写入日志
- `DEK` 明文落库
- `KEK` 落库
- 在调试接口返回私钥或包装材料
- 用 Owner 密码直接加密钱包私钥

最后一条必须明确禁止，因为：

- Owner 密码是登录凭证，不是根密钥
- 凭证轮换不应触发钱包重加密逻辑

## 10. 访问令牌与权限安全

Owner 登录建议：

- 使用 `Bearer JWT`
- 默认短期访问令牌，建议 `1 小时`
- 不提供 refresh token，过期后重新登录

高风险动作要求：

- 已登录
- 二次确认
- 审计记录

## 11. `KEK` 轮换支持边界

本期不提供常规 UI / HTTP API 的 `KEK` 轮换入口，但核心层必须支持：

- 使用当前 `KEK` 解包已存储的 `DEK`
- 使用新 `KEK` 重新包装 `DEK`
- 原子更新钱包中保存的包装后 `DEK`
- 同步更新系统配置快照中的 `kekProvider` 与 `kekReference`
- 记录系统级审计日志

推荐形态：

- 应用层提供独立的 `KEK` 重包装能力
- 部署 / 运维侧通过维护脚本或运维流程触发
- 不作为普通 Owner 能力暴露

## 12. 安全结论

本期安全架构的核心原则是：

`私钥由系统使用，但根密钥由部署侧掌握。`
