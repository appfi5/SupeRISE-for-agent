# 钱包完整重构需求文档（Node MVP）

> 状态：Draft
> 更新时间：2026-03-06
> 文档目标：基于当前仓库实现，整理本次完整重构的 MVP 需求，为后续架构与实施提供统一基线。

## 1. 背景

当前项目的钱包能力分散在两个运行时中：

- `SupeRISELocalServer`：.NET 本地签名服务，负责私钥存储、消息签名、CKB 交易签名、简单配置 UI。
- `SupeRISECli`：Bun/TypeScript CLI，负责 CKB 转账、地址簿、市场认证、sustain 流程，以及现有 MCP 工具。

这种结构在早期可用，但已经出现明显问题：

- 运行时分裂：钱包主能力分布在 .NET 与 Bun 两套实现中。
- 逻辑分散：交易构建在 CLI，签名在 local server，agent 交互又走另一套 MCP。
- 扩展困难：后续要支持 `ETH` 时，继续在多处复制链逻辑的成本会持续上升。
- 部署复杂：当前更像“本地 signer + 本地 CLI”的组合，不适合直接作为统一的 Docker 化钱包服务交付。

因此，本次重构不是迁移某个模块，而是**围绕钱包域做完整重构**。

## 2. 当前实现基线

本次需求文档以当前仓库已有能力为基线，不从零抽象业务。

### 2.1 当前已存在的钱包能力

| 能力域 | 当前实现 | 现状说明 |
| --- | --- | --- |
| 私钥导入与存储 | .NET LocalServer | 支持手工录入私钥并持久化到 SQLite |
| CKB 消息签名 | .NET LocalServer + CLI | CLI 调 signer 服务发起签名 |
| CKB 交易签名 | .NET LocalServer + CLI | CLI 构建 unsigned tx，signer 负责签名 |
| CKB 转账 | CLI | 已具备构建、签名、广播的完整转账链路 |
| 钱包身份查询 | CLI + signer | 已有 `whoami` / `key-config/list` 能力 |
| 地址簿 | CLI | 本地联系人管理，当前与 signer 解耦 |
| 市场认证 | CLI | 已有基于 CKB 地址签名的 market auth 流程 |
| MCP | CLI | 当前 MCP 仅覆盖 sustain / market 平台交互，不直接覆盖钱包域 |

### 2.2 当前实现暴露出的结构问题

- 钱包相关逻辑没有统一收敛到单一服务边界。
- 签名能力与交易构建能力被拆到不同运行时，后续支持多链会持续放大复杂度。
- MCP 没有直接面向钱包域，agent 无法通过统一钱包接口完成核心操作。
- 当前工程仍保留 `.NET + Bun` 的双运行时心智模型，与本次“完全采用 Node”的方向冲突。

## 3. 本次重构目标

本次重构的目标如下：

1. 完全采用 `Node.js` 作为钱包系统运行时，不再保留 `.NET` signer 作为核心实现。
2. 将钱包域能力统一收敛为一个 Node 钱包服务，避免相同逻辑继续散落在 CLI、local server、MCP 中。
3. 以当前 `CKB` 能力为 MVP 交付基线，保证现有核心链路可在新系统内闭环。
4. 在架构层面为 `ETH` 预留扩展位，但 `ETH` 不作为本次 MVP 的交付阻塞项。
5. 支持 Docker 一键部署，降低部署和运行门槛。
6. 提供面向 agent 的 MCP 工具，使 agent 能通过统一入口与钱包交互。
7. 私钥安全按 `MVP` 标准处理：要明确风险边界，但暂不追求 HSM/KMS 级别安全性。
8. 现有 `sustain` 功能先迁移到独立子包中，保持现有行为，后续再单独重构。
9. 允许破坏性改动，不以兼容旧实现为目标，但必须保证旧钱包能力在新系统中有明确对应实现。

## 4. 产品定位

本次重构后的钱包系统定位为：

- 一个**单租户、自托管、Node.js 实现的钱包服务**
- 面向两类使用者：
  - 人类操作者：通过 HTTP API、后续可能的管理 UI 或 CLI 操作钱包
  - Agent：通过 MCP 工具间接调用钱包服务

该系统的职责是：

- 管理私钥与钱包元数据
- 提供链相关签名与转账能力
- 为外部系统（agent、市场集成、后续 CLI/UI）提供统一钱包接口

## 5. 范围

### 5.1 本次 MVP 范围内

- Node.js 钱包 API 服务
- SQLite 持久化
- CKB 钱包能力完整闭环
- 私钥导入、查看钱包身份、消息签名、CKB 转账
- 私钥导出能力（满足 owner 可恢复控制权）
- 面向 agent 的 MCP 工具
- Docker / Docker Compose 一键部署
- 为 ETH 预留链适配扩展位
- 将现有 `sustain` 代码迁移到独立子包，先完成结构解耦

### 5.2 本次 MVP 范围外

- HSM / KMS / TPM / 硬件钱包集成
- 高安全等级的不可导出密钥模型
- 多租户钱包托管平台
- 完整 Web 管理后台
- ETH 交易完整交付
- 与旧 `.NET` API 的兼容适配层
- `sustain` 业务规则、平台集成与 MCP 重构

## 6. 关键假设

为避免需求虚化，本次文档明确采用以下假设：

1. 本次交付目标是 `MVP`，优先保证功能统一和交付速度，不追求最高安全等级。
2. 服务默认面向单实例、单 owner 使用场景。
3. 当前最重要的链是 `CKB`，`ETH` 作为下一阶段扩展目标。
4. 现有 `sustain` 功能在本期只做目录级迁移，不做业务重构。
5. 现有 sustain / market 相关流程不在本次钱包重构的主交付范围内，但新钱包服务必须能为其提供后续集成能力。
6. Agent 与钱包服务的交互以 `MCP` 为主，MCP 自身不承载钱包核心逻辑，只负责调用钱包服务。
7. 如果旧行为与本次需求文档冲突，以本次需求文档为准。

## 7. 功能需求

### 7.1 平台与运行时

**FR-PLAT-1**：系统运行时统一为 `Node.js LTS`。  

**FR-PLAT-2**：项目不再依赖 `.NET` 运行时提供钱包核心能力。  

**FR-PLAT-3**：钱包核心逻辑必须有单一实现，不允许再出现“API 一套、CLI 一套、MCP 一套”各自维护业务逻辑的情况。  

**FR-PLAT-4**：代码仓库采用 `pnpm workspaces` 组织，钱包 API、钱包 MCP、CLI 外壳、钱包核心包与 sustain 子包全部收敛到同一 Node monorepo。  

### 7.2 钱包与密钥管理

**FR-KEY-1**：支持导入私钥作为钱包来源。MVP 至少支持导入单个原始私钥。  

**FR-KEY-2**：导入后的私钥需要加密存储到本地数据库，不允许明文持久化。  

**FR-KEY-3**：支持查询钱包列表、当前默认钱包、钱包基础元数据。  

**FR-KEY-4**：支持 owner 导出私钥，避免资产控制权被系统永久锁定。  

**FR-KEY-5**：导出能力在 MVP 中可以采用简化方案，但必须有显式鉴权、审计或风险提示，不能做成匿名能力。  

**FR-KEY-6**：MVP 不要求真正的 non-exportable key，也不宣称具备硬件级私钥保护。  

### 7.3 CKB 能力

**FR-CKB-1**：支持根据已导入钱包计算并返回 CKB 地址、公钥等身份信息。  

**FR-CKB-2**：支持 CKB 消息签名，以兼容当前市场认证流程。  

**FR-CKB-3**：支持 CKB unsigned transaction 的签名。  

**FR-CKB-4**：支持 CKB 转账闭环能力，至少包含：

- 构建转账交易
- 签名交易
- 广播交易
- 返回 tx hash 与关键执行结果

**FR-CKB-5**：转账相关链逻辑必须统一归属于新钱包服务，不能继续保留“客户端构建交易、服务端只签名”的双端拆分模式。  

### 7.4 ETH 扩展约束

**FR-ETH-READY-1**：系统架构必须预留链适配抽象，保证后续增加 ETH 时无需重写存储模型、鉴权模型与 MCP 交互模型。  

**FR-ETH-READY-2**：链能力应通过链适配器扩展，而不是复制一套独立钱包服务。  

**FR-ETH-READY-3**：本次 MVP 可以不交付完整 ETH 能力，但架构文档必须覆盖 ETH 接入方式。  

### 7.5 API 服务

**FR-API-1**：提供统一 HTTP API 作为钱包服务主入口。  

**FR-API-2**：MVP API 需要覆盖以下能力域：

- 钱包导入与导出
- 钱包身份查询
- CKB 消息签名
- CKB 转账
- 健康检查

**FR-API-3**：API 返回格式统一为 JSON，便于 CLI、MCP、后续 UI 复用。  

**FR-API-4**：MVP 需要提供最小可用鉴权方案，至少区分管理类操作与普通查询/签名类操作。  

### 7.6 MCP 工具

**FR-MCP-1**：提供一个面向 agent 的 MCP server。  

**FR-MCP-2**：MCP 工具覆盖的钱包域能力至少包括：

- 查询当前钱包身份
- 查询钱包列表
- CKB 消息签名
- CKB 转账
- 导入私钥
- 导出私钥

**FR-MCP-3**：MCP 仅作为交互适配层，不直接访问数据库，不自行实现钱包业务规则。  

**FR-MCP-4**：MCP 与钱包服务的交互应基于统一 API 或统一 SDK，保证 agent 调用链与外部系统调用链一致。  

### 7.7 Sustain 迁移

**FR-SUSTAIN-1**：现有 `sustain` 功能从 CLI 内部目录迁移到独立子包。  

**FR-SUSTAIN-2**：本期仅做结构迁移，不调整 `sustain` 现有业务规则、平台流程和命令语义。  

**FR-SUSTAIN-3**：`sustain` 后续改造所需的钱包能力统一改为调用新钱包 API 或钱包 SDK，但这部分适配不作为本次 MVP 验收前置条件。  

### 7.8 Docker 部署

**FR-DEPLOY-1**：支持 Docker 一键部署。  

**FR-DEPLOY-2**：支持 `docker compose up -d` 级别的启动体验。  

**FR-DEPLOY-3**：钱包数据库和运行数据需要持久卷。  

**FR-DEPLOY-4**：必须提供示例环境变量配置，至少包括：

- API 监听地址与端口
- SQLite 数据路径
- CKB RPC / Indexer 配置
- 私钥加密主密钥或主密码来源
- 管理鉴权配置

**FR-DEPLOY-5**：提供健康检查接口，方便容器编排和运行探活。  

## 8. 非功能需求

### 8.1 可维护性

**NFR-MAIN-1**：钱包核心逻辑必须集中，不允许不同入口重复实现。  

**NFR-MAIN-2**：链相关逻辑必须模块化，CKB 与 ETH 通过统一扩展机制接入。  

**NFR-MAIN-3**：新架构要便于后续增加 Web UI、CLI 或更多 agent 集成。  

### 8.2 安全边界（MVP）

**NFR-SEC-1**：私钥需要加密落盘。  

**NFR-SEC-2**：MVP 不要求 HSM/KMS，也不要求进程内达到高安全内存隔离。  

**NFR-SEC-3**：文档必须明确说明：该版本属于软件托管私钥的 MVP，不适用于高价值资产的强安全托管。  

### 8.3 可部署性

**NFR-OPS-1**：默认部署方式为容器化。  

**NFR-OPS-2**：服务启动与配置应尽量简化，避免要求操作者额外安装 `.NET`、Bun 或多套运行时。  

### 8.4 兼容与迁移

**NFR-MIG-1**：本次重构允许破坏性改动。  

**NFR-MIG-2**：不要求兼容旧 `.NET` API、旧 CLI 命令名、旧存储结构或旧部署结构。  

**NFR-MIG-3**：验收以钱包能力对齐为准，不以接口兼容性为准。  

**NFR-MIG-4**：现有项目中的 CKB 身份、签名、转账、market auth 依赖链路需要能在新钱包服务中找到明确替代能力。  

**NFR-MIG-5**：如果旧实现行为与本次需求文档冲突，以本次需求文档为准。  

**NFR-MIG-6**：现有 `sustain` 功能在迁移到子包后，行为保持不变，不在本期引入新的业务回归。  

## 9. MVP 交付物

本次重构完成时，应至少交付以下内容：

1. 一个可运行的 Node 钱包 API 服务
2. 一个可供 agent 使用的 MCP server
3. 一套 Docker / Docker Compose 部署文件
4. 一套面向 CKB 的钱包核心能力
5. 一份架构文档，说明后续如何接入 ETH
6. 一个独立的 `sustain` 子包，承接现有 sustain 代码

## 10. 验收口径

满足以下条件时，可认为本次 MVP 需求完成：

1. 使用 Docker 能在全新环境中启动钱包服务。
2. 能通过 API 导入私钥并查询对应 CKB 身份。
3. 能通过 API 和 MCP 完成 CKB 消息签名。
4. 能通过 API 和 MCP 完成一次 CKB 转账闭环。
5. 能由 owner 触发私钥导出。
6. 不再依赖 `.NET` signer 才能完成钱包核心链路。
7. 钱包核心逻辑只保留一套 Node 实现。
8. 现有 `sustain` 代码已迁移到独立子包，且行为保持不变。
